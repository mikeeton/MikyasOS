# mikyasOS Constitution

Status: Immutable Product And Engineering Standard

## Purpose

The mikyasOS Constitution defines the rules every engineer, AI coding assistant,
designer, architect, product manager, and contributor must follow.

These principles override convenience, speed, and personal preference. If an
implementation violates the Constitution, the implementation must change.

## Immutable Principles

1. **The user comes first.** Optimise for people running businesses.
2. **Consistency beats novelty.** Reuse existing UI, motion, layout, and component
   systems.
3. **AI is infrastructure.** AI belongs across modules as explanation, prediction,
   recommendation, automation, and decision support.
4. **Every feature must feel complete.** Loading, empty, error, permission, and recovery
   states are part of the feature.
5. **Enterprise first.** Design for organisations that may grow into thousands of users
   and millions of records.
6. **Every module belongs together.** CRM, Projects, Finance, Documents, AI, Automation,
   Analytics, Integrations, and Administration must feel like one operating system.
7. **Performance is a feature.** Latency, bundle size, search speed, animation smoothness,
   and dashboard responsiveness affect trust.
8. **Security is mandatory.** Authentication, permissions, validation, encryption, audit,
   and tenant isolation are never optional.
9. **Accessibility is mandatory.** Keyboard support, screen readers, reduced motion, high
   contrast, touch, and responsive layouts are quality requirements.
10. **Design must age well.** Avoid gimmicks and visual trends that weaken long-term trust.
11. **Build platforms, not isolated features.** Prefer reusable systems, APIs, workflows,
    components, AI capabilities, and automation patterns.
12. **Every screen answers three questions.** Where am I? What can I do? What should I do
    next?
13. **Everything connects.** Important objects should support timeline, comments, files,
    notifications, AI, automation, search, permissions, and analytics.
14. **Reduce cognitive load.** Users should think about their business, not the software.
15. **Every click has value.** Remove unnecessary steps and unnecessary information
    requests.
16. **Measure everything.** Usage, performance, errors, business value, AI effectiveness,
    and automation impact should be observable.
17. **Code must be readable.** Prefer simple code, predictable architecture, meaningful
    names, small functions, and useful documentation.
18. **Refactor continuously.** Each release should leave the codebase cleaner.
19. **Never surprise users.** Navigation, permissions, actions, animations, and search
    should behave predictably.
20. **The product should feel alive.** Live updates, meaningful notifications, context
    awareness, and grounded AI recommendations should make the platform feel intelligent.

## Engineering Oath

Every contributor agrees:

- I will leave mikyasOS better than I found it.
- I will not sacrifice quality for speed.
- I will build reusable systems.
- I will respect the design language.
- I will maintain consistency.
- I will think about enterprise scale.
- I will protect customer trust.
- I will improve the user experience.
- I will write software I am proud of.

## AI Coding Assistant Rules

Before making code changes:

1. Read the existing architecture.
2. Preserve consistency.
3. Avoid unnecessary refactoring.
4. Reuse existing components.
5. Follow design tokens.
6. Follow accessibility standards.
7. Follow security standards.
8. Test meaningful changes.
9. Document significant changes.

Never make sweeping architectural changes unless explicitly requested. Always improve the
codebase without breaking existing functionality.

## Final Standard

Every change should make mikyasOS smarter, simpler, faster, more beautiful, more reliable,
more secure, more maintainable, more scalable, and more enjoyable.

## Pledge

mikyasOS is being built to become the operating system businesses trust every day. Every
line of code, interaction, AI recommendation, workflow, and decision should move the
product closer to that vision.
